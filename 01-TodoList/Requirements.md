# Requirements (요구사항 정의)

## 📋 프로젝트 개요

TodoList API는 사용자가 할 일을 생성, 조회, 수정, 삭제할 수 있는 RESTful API입니다.

---

## 🎯 기능 요구사항 (Functional Requirements)

### 1. Todo 생성 (Create)
- **FR-01**: 사용자는 제목과 설명을 포함한 새로운 Todo를 생성할 수 있다
- **FR-02**: Todo 생성 시 고유한 ID가 자동으로 부여된다
- **FR-03**: Todo 생성 시 생성 시간(createdAt)이 자동으로 기록된다
- **FR-04**: 새로 생성된 Todo의 완료 상태(completed)는 기본적으로 false이다

### 2. Todo 목록 조회 (Read All)
- **FR-05**: 사용자는 모든 Todo 목록을 조회할 수 있다
- **FR-06**: Todo 목록은 생성 시간 기준 최신순으로 정렬된다
- **FR-07**: 빈 목록인 경우 빈 배열([])을 반환한다

### 3. 특정 Todo 조회 (Read One)
- **FR-08**: 사용자는 ID를 통해 특정 Todo의 상세 정보를 조회할 수 있다
- **FR-09**: 존재하지 않는 ID 조회 시 적절한 에러 메시지를 반환한다

### 4. Todo 수정 (Update)
- **FR-10**: 사용자는 Todo의 제목, 설명, 완료 상태를 수정할 수 있다
- **FR-11**: 수정 시 수정 시간(updatedAt)이 자동으로 갱신된다
- **FR-12**: 존재하지 않는 Todo 수정 시도 시 에러를 반환한다
- **FR-13**: 부분 수정(Partial Update)을 지원한다

### 5. Todo 삭제 (Delete)
- **FR-14**: 사용자는 ID를 통해 특정 Todo를 삭제할 수 있다
- **FR-15**: 삭제 성공 시 삭제된 Todo 정보를 반환한다
- **FR-16**: 존재하지 않는 Todo 삭제 시도 시 에러를 반환한다

### 6. Todo 완료 상태 토글 (Toggle)
- **FR-17**: 사용자는 Todo의 완료 상태를 쉽게 토글할 수 있다
- **FR-18**: 토글 시 updatedAt이 자동으로 갱신된다

## 🔒 비기능 요구사항 (Non-Functional Requirements)

### 1. 성능 (Performance)
- **NFR-01**: API 응답 시간은 100ms 이내여야 한다
- **NFR-02**: 동시에 최소 100개의 요청을 처리할 수 있어야 한다

### 2. 보안 (Security)
- **NFR-03**: 모든 입력값은 검증되어야 한다
- **NFR-04**: SQL Injection, XSS 등 보안 취약점이 없어야 한다
- **NFR-05**: 민감한 에러 정보가 클라이언트에 노출되지 않아야 한다

### 3. 안정성 (Reliability)
- **NFR-06**: 예상치 못한 에러 발생 시에도 서버가 중단되지 않아야 한다
- **NFR-07**: 모든 에러는 적절하게 처리되고 로깅되어야 한다

### 4. 유지보수성 (Maintainability)
- **NFR-08**: 코드는 명확한 주석과 함께 작성되어야 한다
- **NFR-09**: 각 모듈은 단일 책임 원칙을 따라야 한다
- **NFR-10**: 코드 스타일은 일관성을 유지해야 한다

### 5. 확장성 (Scalability)
- **NFR-11**: 새로운 기능 추가가 용이한 구조여야 한다
- **NFR-12**: 데이터 저장소 변경이 쉬워야 한다 (파일 → DB)

## 📊 데이터 요구사항

### Todo 엔티티
```javascript
{
  id: string,           // UUID v4 형식
  title: string,        // 1-100자, 필수
  description: string,  // 0-500자, 선택
  completed: boolean,   // true/false, 필수
  createdAt: string,    // ISO 8601 형식
  updatedAt: string     // ISO 8601 형식
}
```

### 검증 규칙
- **title**: 
  - 필수 항목
  - 1자 이상 100자 이하
  - 공백만 있는 경우 불가
  
- **description**: 
  - 선택 항목
  - 최대 500자
  
- **completed**: 
  - boolean 타입만 허용
  - 기본값: false

## 🚫 제약사항 (Constraints)

### 기술적 제약
- **TC-01**: Vanilla 버전은 외부 라이브러리 없이 순수 Node.js만 사용
- **TC-02**: 데이터는 메모리 또는 파일 시스템에 저장 (초기 버전)
- **TC-03**: Node.js 18 이상 버전 사용

### 비즈니스 제약
- **BC-01**: 한 사용자가 생성할 수 있는 Todo 개수 제한 없음
- **BC-02**: Todo ID는 중복될 수 없음
- **BC-03**: 삭제된 Todo는 복구 불가능

## ✅ 성공 기준 (Acceptance Criteria)

### 개발 완료 기준
1. ✅ 모든 CRUD 기능이 정상 동작
2. ✅ API 명세서대로 정확한 응답 반환
3. ✅ 모든 에러 케이스 처리
4. ✅ 코드에 충분한 주석 작성
5. ✅ 입력값 검증 완료

### 코드 품질 기준
1. ✅ 각 함수는 50줄 이내
2. ✅ 중복 코드 없음
3. ✅ 명확한 변수/함수명 사용
4. ✅ 에러 처리 일관성 유지
